# 1.1.2 构建系统概览

LLVM的构建系统是其工程实践的重要组成部分，体现了大型C++项目的构建最佳实践。理解这个系统不仅能帮助我们正确构建LLVM，更能学习到工业级项目的构建配置策略。

## 为什么选择CMake？

LLVM使用CMake作为其构建系统，这个选择不是偶然的。从 [llvm/CMakeLists.txt:2](/llvm/CMakeLists.txt#L2) 可以看到最低版本要求是3.20.0，但为什么选择CMake而不是其他构建系统？

### CMake的优势

CMake之所以成为LLVM的选择，源于其多方面的技术优势。在跨平台支持方面，CMake可以生成针对不同平台的构建文件，包括Makefiles、Ninja、Visual Studio、Xcode等，这对于需要支持多平台的编译器基础设施至关重要。成熟的生态系统是另一个关键因素，CMake拥有丰富的模块和工具链支持，特别适合像LLVM这样需要复杂配置的项目。尽管CMake语言有其特殊性，但它足够灵活来表达LLVM复杂的构建需求。更重要的是，CMake已经成为C++项目的事实标准，这大大降低了新贡献者的学习门槛。

### 版本选择策略

从 [llvm/CMakeLists.txt:2](/llvm/CMakeLists.txt#L2) 可以看到，LLVM要求的最低CMake版本是3.20.0。选择3.20.0作为最低版本是一个经过权衡的决策。这个版本足够新，引入了一些有用的新特性，可以简化构建脚本；同时又足够老，大多数开发环境都能满足这个要求。这种策略在新特性可用性和广泛部署之间取得了平衡，既不过于保守而失去新特性，也不过于激进而要求用户频繁更新CMake。

## 源外构建的设计理念

LLVM强烈推荐使用源外构建（out-of-source build）方式。这不是一个随意的建议，而是基于多年工程实践得出的最佳实践。

### 为什么源外构建如此重要？

源外构建的第一个核心优势是保持源码目录干净。构建产生的临时文件（.o、.a、.so等）不会出现在源码目录，从而避免被误提交到版本控制。这使得源码目录的变化都是真正的代码变更，不会被构建产物干扰。在极端情况下，甚至可以将源码目录设为只读，确保构建过程不会意外修改源码。

支持多种构建配置是另一个重要优势。开发者可以同时维护Debug和Release两个独立构建，无需重新配置；也可以为不同目标架构创建独立的构建目录；还可以创建实验性的构建配置，而不影响正常的开发构建。这种灵活性对于编译器开发尤为重要。

源外构建还使清理工作变得异常简单。只需删除构建目录即可清理所有构建产物，不用担心某些隐藏的构建文件残留导致奇怪的构建问题，也可以轻松释放不再需要的构建占用的磁盘空间。

最后，源外构建避免了版本控制干扰。不需要在.gitignore中添加大量构建产物模式，git status只显示真正的源码变更，使用git clean也不会意外删除重要的构建文件。

### 源码内构建为什么被禁止？

从历史上看，LLVM曾经支持源码内构建，但后来完全禁止了。这是因为源码内构建容易产生循环依赖和错误的依赖关系，当构建出现问题时，很难区分是源码问题还是构建产物问题。在持续集成环境中，源码内构建还会导致更多的状态残留问题。

## CMake参数的深入理解

让我们分析LLVM构建中常用的CMake参数及其设计意图。

### 生成器选择（-G）

```bash
cmake -G Ninja -S ../llvm -B .
```

在生成器选择方面，Ninja通常比Make更适合LLVM的构建。Ninja在并行任务调度方面比Make更智能，增量构建通常比Make快2-10倍，对依赖关系的处理也更精确，而且Ninja的构建文件更简洁，易于调试。当然，Make在某些场景下仍有其优势，它具有更广泛的兼容性，几乎所有Unix系统都有Make；调试输出有时更容易理解；对某些特殊构建规则的支持也更成熟。

### 源码和构建目录（-S、-B）

```bash
cmake -S ../llvm -B .
```

这种明确指定源目录和构建目录的方式是CMake 3.13+推荐的方式。它具有自文档化的优点，命令本身就说明了哪个是源目录，哪个是构建目录；同时减少了在错误目录运行CMake的风险，并且可以从任何位置运行CMake命令。

## C++标准要求的演变

从 [llvm/CMakeLists.txt:62](/llvm/CMakeLists.txt#L62) 可以看到，LLVM当前要求的C++标准是C++17。LLVM对C++标准的选择是一个渐进的过程，LLVM 10及更早版本使用C++11，LLVM 11-13使用C++14，从LLVM 14开始升级到C++17。

选择C++17而不是更新的标准是基于多方面的考虑。C++17在所有主流编译器（GCC、Clang、MSVC）中都得到了良好支持，而且已经足够成熟，没有C++20那么多的实现差异。同时，C++17提供了足够有用的特性，如结构化绑定、if constexpr、fold表达式等。当然，选择C++17也考虑到了用户可能还在使用较老的编译器这一现实情况。

值得注意的是，用户可以选择更新的标准来构建LLVM，只需在CMake命令中添加`-DCMAKE_CXX_STANDARD=20`即可。这是因为LLVM的代码已经考虑了前向兼容性。

## 构建类型的设计考量

LLVM利用CMake的标准构建类型，但针对编译器工作负载进行了调整。

### Debug构建

```bash
cmake -DCMAKE_BUILD_TYPE=Debug -DLLVM_ENABLE_ASSERTIONS=ON -S ../llvm -B .
```

Debug构建适用于日常开发、问题调试和学习探索场景。在设计权衡上，Debug构建使用-O0无优化，生成的代码最接近源码；包含-g完整调试信息；启用LLVM_ENABLE_ASSERTIONS=ON以捕获更多错误。由于没有优化，编译速度相对较快。

### Release构建

```bash
cmake -DCMAKE_BUILD_TYPE=Release -DLLVM_ENABLE_ASSERTIONS=OFF -S ../llvm -B .
```

Release构建适用于生产使用、基准测试和发布版本。它使用-O3最高级别优化，不包含调试信息，禁用断言以减少运行时开销。当然，优化过程需要更多时间，编译时间相对较长。

### RelWithDebInfo构建

```bash
cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo -S ../llvm -B .
```

RelWithDebInfo构建适用于性能分析、生产环境调试以及需要在性能与调试之间取得平衡的场景。它使用-O2较高优化级别，同时保留-g调试信息。虽然调试信息增加了二进制大小，但这是性能问题定位的最佳选择。

### MinSizeRel构建

```bash
cmake -DCMAKE_BUILD_TYPE=MinSizeRel -S ../llvm -B .
```

MinSizeRel构建主要针对嵌入式环境等空间受限的场景，以及需要减少下载大小的分发优化。它使用-Os优化代码尺寸，不包含调试信息以最小化尺寸，但某些优化可能为了尺寸而牺牲速度。

## 并行编译的性能考量

```bash
cmake --build . -j$(nproc)
```

选择合适的并行任务数是一个权衡过程。

### 为什么不总是用$(nproc)？

尽管使用全部处理器核心看起来是最优选择，但实际上存在几个限制因素。首先是内存限制，每个编译任务都需要一定内存，内存不足会导致交换甚至OOM。其次，链接阶段通常是单线程的，太多并行任务会造成资源浪费。最后，太多编译任务可能导致磁盘I/O成为瓶颈。

### 推荐策略

根据系统内存的不同，推荐的并行数也有所差异。对于8GB内存的系统，建议使用-j4作为保守策略，避免OOM；16GB内存的系统可以使用-j8，这在大多数情况下都能达到良好平衡；对于32GB及以上内存的系统，可以使用-j$(nproc)充分利用全部核心。

### 任务数的动态调整

有时可以在不同阶段使用不同的并行数。在编译阶段使用较多并行，比如-j16；如果遇到内存不足的情况，可以重新运行时使用较少并行，比如-j4。这种动态调整策略可以充分利用系统资源，同时避免因内存不足导致的构建失败。

## 增量构建的工作原理

CMake的增量构建是一个智能的过程，值得深入理解。

### 依赖跟踪

CMake通过多种方式跟踪依赖关系。首先是文件时间戳，比较源文件和目标文件的时间戳来判断是否需要重新构建。其次是头文件依赖，自动检测#include的头文件变化。最后是配置依赖，当CMakeLists.txt变化时会重新配置整个项目。

### 常见问题及解决方案

在使用增量构建时可能会遇到一些问题。第一个常见问题是修改了头文件但没有重新编译，这通常是因为构建系统没有正确跟踪到头文件依赖。解决方案是强制重新生成依赖信息，运行`cmake .`然后重新构建。

第二个问题是CMakeLists.txt修改后没有生效，这是因为需要重新运行CMake配置。可以直接运行`cmake .`重新配置，或者完全重新配置，删除构建目录的所有内容后重新运行CMake命令。

第三个问题是增量构建太慢。可能的原因包括头文件被过度包含、预编译头没有正确使用或依赖图过于复杂。相应的解决方案包括检查是否有不必要的#include、使用前向声明减少头文件依赖，以及考虑使用C++20的模块特性。

## 构建配置的常见场景

### 场景1：新贡献者的首次构建

```bash
cmake -G Ninja \
  -DCMAKE_BUILD_TYPE=Debug \
  -DLLVM_ENABLE_ASSERTIONS=ON \
  -DLLVM_TARGETS_TO_BUILD="X86" \
  -DLLVM_ENABLE_PROJECTS="clang" \
  -S ../llvm \
  -B build
```

对于新贡献者的首次构建，选择Debug模式便于调试和学习，启用断言可以快速发现问题，仅构建X86目标可以减少编译时间，包含Clang作为最常用的子项目。这种配置在体验和效率之间取得了良好平衡。

### 场景2：性能调优

```bash
cmake -G Ninja \
  -DCMAKE_BUILD_TYPE=RelWithDebInfo \
  -DLLVM_ENABLE_ASSERTIONS=ON \
  -DLLVM_TARGETS_TO_BUILD="X86;RISCV" \
  -S ../llvm \
  -B build
```

对于性能调优场景，RelWithDebInfo模式可以profile优化后的代码，同时断言仍能捕获问题。构建多个目标（如X86和RISCV）可以测试不同目标架构的性能表现。

### 场景3：生产发布

```bash
cmake -G Ninja \
  -DCMAKE_BUILD_TYPE=Release \
  -DLLVM_ENABLE_ASSERTIONS=OFF \
  -DLLVM_TARGETS_TO_BUILD="all" \
  -DLLVM_ENABLE_PROJECTS="clang;lld;mlir;polly" \
  -DCMAKE_INSTALL_PREFIX=/opt/llvm \
  -S ../llvm \
  -B build
```

生产发布配置使用Release模式以获得最佳性能，禁用断言以最小化运行时开销。构建所有目标以支持所有架构，包含多个子项目以提供完整工具链。指定安装路径便于打包分发。

## 常见问题的深层分析

### 编译器检测问题

一个常见问题是CMake找不到合适的编译器。可能的原因包括编译器不在PATH中、版本太旧，或者混合使用不同编译器编译的二进制可能有ABI兼容性问题。

解决方案是显式指定编译器，可以通过环境变量CC和CXX来设置，例如`CC=clang CXX=clang++ cmake -G Ninja -S ../llvm -B .`，或者使用绝对路径`CC=/usr/bin/clang-15 CXX=/usr/bin/clang++-15 cmake -G Ninja -S ../llvm -B .`。

显式指定编译器有几个重要原因：避免使用可能太旧的系统默认编译器，确保所有部分使用相同的编译器以保证一致性，以及锁定特定编译器版本避免意外升级。

### 内存不足问题

构建过程中出现OOM（内存不足）是另一个常见问题。其深层原因包括链接阶段内存峰值，链接器在链接大二进制时需要大量内存；调试信息膨胀，Debug构建的调试信息会显著增加内存使用；以及过度并行导致的内存竞争。

分阶段解决方案是先只编译不链接，使用`cmake --build . -j$(nproc) --target all-but-link`，然后在链接阶段使用少量并行，如`cmake --build . -j2`。或者简单地减少并行数，直接使用`cmake --build . -j2`。

### 目标选择策略

选择要构建的目标是一个重要的决策。可以仅构建X86以获得最快的构建速度，使用`-DLLVM_TARGETS_TO_BUILD="X86"`；或者构建常用目标如X86、ARM、AArch64和RISCV；或者构建所有目标，使用`-DLLVM_TARGETS_TO_BUILD="all"`。

选择建议是在开发阶段只构建目标架构，在测试阶段构建几个有代表性的目标，在发布阶段构建所有目标。这种策略可以在开发效率和测试覆盖之间取得平衡。

## 总结

LLVM的构建系统体现了几个核心设计原则：灵活性，支持多种配置和场景；性能优先，优化构建速度和最终代码性能；用户友好，提供合理的默认值和清晰的文档；可扩展性，易于添加新的目标和子项目。理解这些设计原则，能帮助我们更高效地使用和修改LLVM的构建系统。

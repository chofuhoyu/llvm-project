# 1.1.1 目录结构详解

LLVM项目的目录结构不是随意安排的，而是经过深思熟虑的设计，反映了编译器基础设施的架构原则。理解这个组织结构，不仅能帮助我们快速找到需要的代码，更能揭示LLVM的模块化设计思想。

## 顶层目录结构的设计逻辑

当我们查看 [llvm/](/llvm/) 目录时，会发现其组织方式遵循几个重要的软件工程原则。让我们分析每个目录存在的意义及其设计动机。

```
llvm/
├── include/llvm/       # 公共头文件
├── lib/                # 库实现代码
├── tools/              # 独立工具
├── utils/              # 工具脚本和TableGen
├── cmake/              # CMake模块
├── docs/               # 文档
├── examples/           # 示例代码
├── test/               # 测试用例
├── unittests/          # 单元测试
├── benchmarks/         # 基准测试
├── projects/           # 旧的项目目录（已弃用）
├── runtimes/           # 运行时构建
├── bindings/           # 语言绑定
├── resources/          # 资源文件
├── CMakeLists.txt      # 主CMake构建文件
└── README.txt          # 项目说明
```

这个目录结构体现了几个核心的设计原则，包括接口与实现分离、功能模块化、测试与生产代码分离、工具与库分离，以及构建逻辑独立。`include/`和`lib/`的分离是接口与实现分离原则的典型体现，每个目录对应一个独立的功能模块，`test/`和`unittests/`独立于生产代码，可执行工具在`tools/`而库代码在`lib/`，`cmake/`目录专门存放构建系统逻辑。让我们逐个深入分析这些目录。

## include/llvm/ - 公共头文件的设计

[llvm/include/llvm/](/llvm/include/llvm/) 目录包含了LLVM的所有公共头文件。这不仅仅是一个存放头文件的地方，更是LLVM向外部世界展示的API边界。

### 为什么头文件和实现分离？

这种分离有几个重要的好处。首先是编译速度优化，修改实现文件不需要重新编译依赖头文件的所有代码。其次是API稳定性，公共头文件定义了稳定的接口，实现可以独立演进。此外，这种分离支持模块化编译，每个库可以独立编译然后链接在一起，同时头文件本身也作为API文档，说明如何使用LLVM的功能。

### 子目录的组织逻辑

头文件按功能模块组织成多个子目录，这种组织方式反映了LLVM的内部架构。[ADT/](/llvm/include/llvm/ADT/) 目录包含代数数据类型，LLVM需要比STL更高性能或更特殊的容器，因此实现了自己的一套。[IR/](/llvm/include/llvm/IR/) 是中间表示核心，这是LLVM的心脏，定义了IR的核心数据结构。[Analysis/](/llvm/include/llvm/Analysis/) 提供分析框架，定义各种程序分析Pass的接口。[CodeGen/](/llvm/include/llvm/CodeGen/) 是代码生成基础设施，提供从IR到机器码转换的抽象层。[MC/](/llvm/include/llvm/MC/) 是机器码层，提供汇编器、反汇编器的抽象，与具体目标解耦。[Support/](/llvm/include/llvm/Support/) 是支持库，提供平台抽象层，隐藏操作系统差异。[Target/](/llvm/include/llvm/Target/) 是目标架构抽象，定义所有目标后端都需要实现的接口。[Transforms/](/llvm/include/llvm/Transforms/) 提供优化变换，定义IR优化Pass的接口。[TableGen/](/llvm/include/llvm/TableGen/) 是TableGen核心，这是LLVM特有的代码生成工具。

### ADT目录的特殊性

[ADT/](/llvm/include/llvm/ADT/) 目录值得特别关注。LLVM为什么不直接使用STL，而是要实现自己的容器？从历史和实践来看，有几个关键原因。编译器对性能极其敏感，LLVM的容器针对特定使用模式进行了优化。LLVM也需要精确控制内存布局，例如SmallVector的内联存储。此外，PointerUnion、PointerIntPair等是STL没有的特殊容器。从历史角度看，LLVM起源时STL还不够成熟，编译器也不够完善。最后，LLVM可以控制自己的ABI，不依赖特定STL实现。

## lib/ - 库实现的组织策略

[llvm/lib/](/llvm/lib/) 目录包含了LLVM所有库的实现代码。其组织结构与`include/llvm/`基本对应，这种对应关系不是偶然的。

### 为什么一一对应？

这种设计有几个重要的优势。首先是易于定位，看到一个头文件就能很容易找到对应的实现。其次是支持模块化编译，每个目录可以编译成独立的库。目录结构也反映了模块之间的依赖关系，使依赖关系变得清晰。最后，这种组织方式有利于团队协作，不同团队可以在不同目录工作，减少冲突。

### Target目录的特殊结构

[lib/Target/](/llvm/lib/Target/) 目录特别值得关注。每个目标架构都有自己的独立目录，例如X86、ARM、AArch64、RISCV等。每个后端独立目录的设计体现了LLVM的目标抽象理念。这种设计提供了良好的隔离性，每个后端独立，修改一个后端不影响其他后端。同时支持可插拔架构，可以选择性地编译某些后端，减少编译时间和二进制大小。所有后端都实现相同的接口，通过Target/目录的抽象层访问，这使得添加新的目标架构变得简单，只需添加新目录即可。

每个目标后端目录通常包含TableGen文件来描述目标架构的指令、寄存器等，指令选择实现将IR映射到目标指令，寄存器分配支持描述寄存器类和分配策略，以及与目标相关的汇编器/反汇编器支持。

## tools/ - 工具与库的分离

[llvm/tools/](/llvm/tools/) 目录包含了多个独立的可执行工具。这种工具与库分离的设计是LLVM架构的重要特点。

### 为什么工具独立？

这种设计有几个重要的考虑。首先是核心功能作为库，可以被多个工具使用，实现了库复用。其次是用户可以选择需要的工具，不需要全部安装，实现了灵活组合。第三是工具可以独立更新，不影响核心库，支持独立演进。最后，每个工具可以独立测试，使测试变得更加友好。

### 工具的设计哲学

LLVM的工具遵循"小而美"的Unix哲学。[llc](/llvm/tools/llc/) 是LLVM静态编译器，专注于IR到机器码的转换。[opt](/llvm/tools/opt/) 是LLVM优化器，专注于IR的分析和变换。[llvm-dis](/llvm/tools/llvm-dis/) 是LLVM反汇编器，专注于位码到文本IR的转换。[llvm-as](/llvm/tools/llvm-as/) 是LLVM汇编器，专注于文本IR到位码的转换。[llvm-link](/llvm/tools/llvm-link/) 是LLVM链接器，专注于IR模块的链接。

每个工具都专注于做好一件事，然后通过组合使用来完成复杂的任务。这与传统的单一大编译器设计形成鲜明对比。

## utils/ - 构建时工具与TableGen

[llvm/utils/](/llvm/utils/) 目录包含各种实用脚本和工具，其中最重要的是TableGen相关的代码。

### TableGen为什么在这里？

TableGen是LLVM中一个非常特殊的工具，值得单独讨论。它是一个构建时工具，在构建LLVM时运行，生成C++代码。它不是最终用户工具，最终用户通常不需要直接使用TableGen。TableGen使用特定的DSL描述目标架构，作为目标描述语言，同时也是一个代码生成器，生成指令选择、寄存器信息等C++代码。

从 [llvm-tblgen.cpp](/llvm/utils/TableGen/llvm-tblgen.cpp) 可以看出，TableGen是一个相对独立的工具，但对LLVM的构建至关重要。

### TableGen的工作原理

TableGen的工作流程体现了元编程的思想。首先是编写.td文件，使用TableGen DSL描述目标架构。然后llvm-tblgen解析.td文件，构建内存中的记录结构。接着不同的Emitter生成不同的C++代码。最后，生成的C++代码被编译进LLVM库。

这种设计使得添加新的目标架构或修改现有架构变得更加容易，不需要手写大量重复的C++代码。

## test/与unittests/ - 测试策略的体现

LLVM有两个独立的测试目录：[test/](/llvm/test/)和[unittests/](/llvm/unittests/)。这种区分反映了LLVM的多层次测试策略。

### 为什么两种测试？

test/目录包含端到端的集成测试，测试整个工具链，而unittests/目录包含细粒度的单元测试，测试单个组件。两个目录使用不同的测试框架，test/使用lit和FileCheck，unittests/使用Google Test。在运行速度上也有所不同，单元测试通常更快，集成测试更全面。

### lit和FileCheck的设计

LLVM的测试工具lit和FileCheck值得特别关注。lit是LLVM集成测试工具，并行运行测试，处理依赖关系。FileCheck是模式匹配工具，验证输出是否符合预期。这种测试框架的设计体现了LLVM对测试的重视，测试代码和生产代码一样，经过精心设计。

## 目录组织原则的深层思考

LLVM的目录组织不仅仅是代码的存放方式，更是其设计哲学的体现。让我们总结这些原则。

### 接口与实现分离

公共头文件在`include/llvm/`，实现在`lib/`对应目录。这种分离保护了API稳定，实现可以修改而接口保持不变，支持模块化编译，修改实现不需要重新编译所有依赖，同时提供了清晰的边界，明确哪些是公共API，哪些是内部实现。

### 按功能模块组织

每个主要功能模块有独立的目录。这种组织实现了高内聚，相关代码放在一起，实现了低耦合，模块之间通过明确的接口交互，同时也易于导航，可以根据功能快速定位代码。

### 目标后端独立

每个目标架构在`lib/Target/`下有独立目录。这种设计实现了可插拔架构，可以选择编译哪些后端，实现了隔离修改，修改一个后端不影响其他后端，同时提供了统一抽象，所有后端实现相同的接口。

### 工具与库分离

库代码在`lib/`，工具在`tools/`。这种分离实现了库复用，核心库可以被多个工具使用，实现了灵活组合，用户可以选择需要的工具，同时支持独立演进，工具和库可以独立更新。

### 测试独立

测试代码与生产代码分离。这种安排避免了污染，测试代码不混入生产代码，支持多种测试策略，可以使用不同的测试框架，同时也有清晰的责任，明确哪些是测试，哪些是生产代码。

## 对修改LLVM的启示

理解LLVM的目录组织结构，对于正确修改LLVM至关重要。

在添加新功能时，新的IR类应该放在`include/llvm/IR/`和`lib/IR/`，新的优化Pass应该放在`include/llvm/Transforms/`和`lib/Transforms/`，新的目标后端应该放在`lib/Target/NewArch/`。

在组织新代码时，应该遵循现有目录结构，保持头文件和实现分离，按功能模块组织。

同时也有一些不该做的事情，不要在公共头文件中暴露过多实现细节，不要随意修改目录结构，不要将工具和库代码混在一起。

LLVM的目录结构是经过多年演进形成的，理解其背后的设计思想，能帮助我们更好地融入这个项目。
